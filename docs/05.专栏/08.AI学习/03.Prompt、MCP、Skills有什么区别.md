## Promot（提示词）

以前大家用 AI，基本都是 “一句话指挥 AI 干活”，但项目一复杂，问题全暴露了，以前AI刚诞生的时候，大家都在互相分享提示词，比如

> 背景是蓝色的海
>
> 光线柔和，穿透云层，透力强

**提示词太长，AI 反而记不住**

上下文堆太多，AI 注意力被分散，经常漏细节、编答案，也就是常说的 “幻觉”。

**写完就废，没法复用**

今天写的代码规范提示词，明天换个项目、换个人用，又得重新改，完全不通用。

**不像工程，更像 “凑活用”**

全靠自然语言堆要求，没有结构、没有标准，团队协作根本没法统一规范。

## MCP（模型上下文协议）

MCP 是 Anthropic（Claude 的母公司）提出的开放协议，定义了 AI 应用如何与外部数据源、工具进行标准化的通信。

```
┌─────────────┐     MCP 协议     ┌──────────────┐
│  AI 客户端   │ ◄──────────────► │  MCP Server  │
│  (Claude等) │                  │  (数据/工具)  │
└─────────────┘                  └──────┬───────┘
                                        │
                          ┌─────────────┼─────────────┐
                          ▼             ▼             ▼
                     ┌────────┐   ┌────────┐   ┌────────┐
                     │ 本地   │   │ 数据库  │   │ API    │
                     │ 文件   │   │       │   │ 服务   │
                     └────────┘   └────────┘   └────────┘
```



## Skills

现在 AI 工程化落地，早就不靠 “超长提示词” 了，而是靠一种更简单、更实用的东西 ——**Skills（技能包）**。



**Skills = 给 AI 定制的专项 “工作手册 + 执行流程”**。

比如：

- 代码审查 Skill
- 数据库慢查询优化 Skill
- 接口自动生成 Skill
- 提交规范检查 Skill



它不是一段临时指令，而是一个**可保存、可复用、可分享**的完整能力包。你可以把它理解成：**把程序员的经验、规范、步骤，写成 AI 能看懂的 SOP**。

它有两个最实用的特点：

1. 按需加载，不占内存
- 平时只存个名字，用到时再把完整规则发给 AI，不浪费算力，响应更快。

2. 结构清晰，人人能维护
- 不用写复杂代码，用普通文档就能写，新人也能看懂、能修改。



从结构来说，是一个 `SKILL.md` 文件，包含元数据（描述什么时候用）和正文（具体的执行 SOP），Skill 的完整目录结构是这样的：


```
skill-name/
├── SKILL.md          # 必需：元数据（何时使用）+ 正文（指令、流程、示例）
├── scripts/          # 可选：可执行脚本（Python/Bash），按需调用
├── references/       # 可选：参考文档，按需读取
└── assets/           # 可选：模板、图片等资源
```





## 总结

| 维度           | **MCP**                      | **Skills**                               | **Prompt**                     |
| :------------- | :--------------------------- | :--------------------------------------- | :----------------------------- |
| **全称**       | Model Context Protocol       | 技能（通常指 Function Calling / Tools）  | 提示词                         |
| **本质**       | **通信协议**（标准）         | **能力模块**（功能）                     | **指令文本**（输入）           |
| **作用**       | 让 AI 能连接外部数据源和工具 | 让 AI 能执行具体操作（如查天气、发邮件） | 告诉 AI "该怎么做" 的指令      |
| **类比**       | USB 接口标准                 | 插在 USB 上的设备（打印机、U盘）         | 你对设备说的"帮我打印这个文件" |
| **谁实现的**   | 由 AI 平台/框架提供          | 开发者编写或平台预置                     | 用户每次输入或系统预置         |
| **是否可复用** | 是（跨应用的标准）           | 是（封装好的功能单元）                   | 部分可复用（模板）             |



| 层次       | 扮演角色 | 具体内容                                                     |
| :--------- | :------- | :----------------------------------------------------------- |
| **Prompt** | 你的指令 | "读取 /var/log/app.log 的最新 100 行，分析错误类型，然后把报告发邮件给 admin@example.com" |
| **MCP**    | 连接通道 | `mcp-server-filesystem` 提供读取 `/var/log/` 的能力          |
| **Skills** | 执行动作 | `send_email(to, subject, body)` 函数负责发邮件               |



从prompt裸奔，到mcp集成工具来解决ai的不稳定性，再到skill解决token爆炸问题



prompt 看作一次性的两个人间的闲聊，时间久了，俩人忘了之前聊过啥了，同一个话题，下次聊天还会重新由某个相同的话头引出。

skills 看作某个东西的使用说明书，规定了你先这样，再那样，保证了正确的使用流程。

mcp则看作一个type-c充电器，充电的输入输出规则固定了，可以给不同型号的支持type-c充电口的手机充电。
